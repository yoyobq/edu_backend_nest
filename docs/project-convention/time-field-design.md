# 时间字段设计规范（简版）

## 1. 统一规则

本项目时间字段只分三类：

* **系统事件时间**：`TIMESTAMP(3)`
* **纯业务日期**：`DATE`
* **业务上的具体日期时间**：`DATETIME`

不单独推广 `TIME` 作为通用规则。

---

## 2. 使用原则

### `TIMESTAMP(3)`

用于表示“某一刻发生了什么”的系统事件时间。

### `DATE`

用于表示“业务上是哪一天”的纯日期字段，只关心日，不关心时分秒。

### `DATETIME`

用于表示“业务上约定的某天某时”的字段，属于业务时间，不等同于系统事件时间。

---

## 3. 禁止事项

* 不要把纯业务日期写成 `TIMESTAMP`
* 不要把 `DATE` 映射为 JS `Date`
* 不要为了省事把所有日期都改成 `DATETIME`
* 不要在业务层手工做时区加减
* 不要依赖默认时区配置

---

## 4. TypeORM 约定

* `TIMESTAMP(3)` 对应 `Date`
* `DATE` 对应 `string`
* `DATETIME` 在模块内统一使用方式，不允许同一语义混用

---

## 5. 系统约定

* MySQL 连接必须显式配置 `timezone`
* 系统内部默认按统一时区口径处理
* 时区转换集中在展示层或统一工具层处理

---

## 6. 一句话判断法

先问自己：

> 这个字段表示的是“某一刻发生了什么”，还是“业务上是哪一天 / 某天某时”？

* 某一刻发生 → `TIMESTAMP(3)`
* 业务上是哪一天 → `DATE`
* 业务上某天某时 → `DATETIME`

---

## 7. 语义声明规则

时间语义由**调用场景显式声明**，不由输入值自动推断。

同一个时间字面量在不同场景可表示不同语义，例如：

* 任务真实开始时间 → 系统事件时间
* 预约发布时间 → 业务上的具体日期时间

因此统一要求：

* 先声明语义，再做时间收敛
* 禁止根据字段名、字符串形态、值内容“猜测语义”
* 禁止在通用层自动改判“系统事件时间 / 业务时间”

---

## 8. 处理边界规则（规则导向）

时间处理按四类职责分离，但本文只定义规则，不规定具体函数设计：

* **解析层**：只负责识别是否可解析，并产出中性结果
* **语义收敛层**：仅按调用方声明的语义收敛
* **格式输出层**：只负责稳定输出，不承担语义判定
* **边界校验层**：只做声明范围内的校验，不扩展为业务编排

统一禁止：

* 不补 `now` / `today` / 零点等业务默认值
* 不“尽量修正”非法输入后放行
* 不读取运行时配置或依赖数据库状态
* 不在该层处理时长语义（时长统一走 `*_ms`）

---

## 9. 区间校验最小原则

通用时间区间校验默认只承诺**顺序关系**（`start <= end`）。

以下规则属于业务约束，应由上层场景自行声明并实现，不应混入通用时间规则：

* 必须同时存在或同时为空
* 结束时间不得晚于当前时间
* 时间范围不得超过 N 天
* 早于业务下限或晚于业务上限的判定

## 11. 接入与兼容策略

对已接入新规范的链路，时间规则的强制入口定义为 `time normalize`，而不是 `DTO` 层。

统一约束如下：

* `DTO` 层可做输入约束，但不是唯一规则入口
* `time normalize` 是语义防线，负责在 `Core` 内执行防御式校验
* 旧链路未接入 `time normalize` 时，保持旧行为兼容
* 新链路接入 `time normalize` 后，按新规则获得语义保障
* 格式输出层只接收 normalize 产出的语义载体，拒绝直接传入 `Date`
* 语义载体是可序列化对象，不是 `Date` 实例，允许跨拷贝传递

对业务 `DATETIME` 的最低防线要求：

* 只接受不带时区的 `datetime string`
* 拒绝 `Date`、`epoch`、带时区字符串
* 不允许通过“调用约定”替代 `normalize` 校验

推荐调用路径：

* `raw input` → `parseTimeInput` → `normalizeSystemEventTime | normalizeBusinessDateTime` → `formatForTimestamp3 | formatForDateTime`
